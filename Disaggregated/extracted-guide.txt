Customers will be deploying Azure Local across multiple racks, at higher node counts, and with a broader range of storage architectures to meet regulatory, operational, and resiliency requirements.To address these scenarios, Azure Local intends to support a set of scalable and modular deployment patterns that span:Singlerack to multirack cluster designsCluster sizes from 1 to 64 nodesConnected and fully disconnected operationsMultiple storage architectures, including hyper-converged and disaggregated modelsSANbased storage using Fibre Channel (FC) and iSCSI transportsMicrosoft SDN for hyperconverged clusters up (to 16 nodes) and external SDN for disaggregated clusters from 1 to 64 nodesThese capabilities enable Azure Local to be deployed in environments where compute, networking, and storage are designed, validated, and scaled independently, while still operating as a single, coherent platform.This document establishes a common architectural foundation and describes the network design, topology, and connectivity requirements for multirack clusters (reinforces single-rack or existing clusters)Hyperconverged Clusters with S2DFor hyperconverged clusters with Storage Spaces Direct, the network uses a straightforward two-switch setup. Two Top-of-Rack switches form an HSRP pair for redundancy and link aggregation, with one active and the other on standby. There is no spine layer or VRF isolation, all traffic flows directly through the switch pair. Tenant and management traffic share a Microsoft SET team, supporting active-standby operation.Topology OverviewThe single-rack topology uses two switches  (48 Port×25GbE + 6×100G ) configured as a HSRP pair. Servers connect using Windows Server SET (Switch Embedded Teaming), which provides switch-independent NIC teaming to the nodes.Each TOR switch connects directly to the data center core routers via uplinks using Layer3 routing. There is no spine layer — the TOR pair is the entire fabric. This keeps the design simple and reduces the number of switches, cables, and failure domains./Figure: Single-Rack Topology —HSRP Pair with SET TeamingRoutingRouting in the single-rack deployment is straightforward: two protocols, two purposes. iBGP synchronizes routes between the TOR pair, and Layer3 routing exchanges routes with the upstream data center core. All forwarding decisions use standard IP routing with SVIs (Switched Virtual Interfaces) on each VLAN.iBGP — TOR-to-TOR Route SyncBoth TOR switches share the same Autonomous System Number (ASN 65001) and run iBGP between them over reserved inter-switch ports (Eth1/41–42). These reserved ports are also used for MLAG control-plane keepalive/heartbeat, keeping switch-to-switch control traffic off of host-facing ports.Layer3 — Uplinks to DC CoreEach TOR switch connects with the data center core routers over uplinks (Eth1/53–54). The core routers advertise a default route (0.0.0.0/0) to the TOR switches, providing internet and corporate network access. In return, the TOR switches advertise their local subnets (management, compute, tenant VLANs) to the core.Because both TORs peer independently with the core, the cluster maintains upstream connectivity even if one TOR or one core router fails. Routing convergence handles the failover automatically.VLANs & SVIsEach TOR switch has SVIs (Layer 3 interfaces) for Management and Compute VLANs, with HSRP providing a shared virtual IP (VIP) as the default gateway. Servers always point to the VIP — they never need to know which physical switch is active. If TOR-A fails, TOR-B takes over the VIP within seconds.VLANNameModePurposeHSRP VIP Example7Infra/MgmtAccessManagement (native)10.0.7.16TenantTrunkTenant VM traffic10.0.6.1201Tenant2TrunkAdditional tenant traffic10.0.201.1711Storage-ATrunkS2D RDMA (TOR-A trunk)N/A (L2 only)712Storage-BTrunkS2D RDMA (TOR-B trunk)N/A (L2 only)25BMCAccessOut-of-band managementN/A (separate switch)Storage VLANs (711 and 712) are Layer 2 only — they do not have SVIs or HSRP because S2D RDMA traffic is switched locally on each TOR. It is not supported to scale out this configuration beyond 16 nodes across multiple racks.Disaggregated Clusters with External StorageFor disaggregated clusters with external storage, Azure Local uses a leaf-spine (Clos) network for better scalability, fewer bottlenecks, and reduced failure impact across racks. This design covers Fibre Channel and iSCSI external storage targets, along with host NIC/HBA settings for connecting to the storage fabric. Disaggregated clusters size can be as small as sing-node and scale up to 64 nodes.The Big Picture — What Are We Building?The ProblemA traditional 2-switch design (one pair of Top-of-Rack switches) works fine for 16 nodes. But at 64 nodes across multiple racks, that model breaks:Port limits: A single TOR switch has 48 ports — you can’t plug 64+ nodes into one pairFailure blast radius: If one switch dies, half your cluster loses connectivityBandwidth bottleneck: All rack-to-rack traffic funnels through a single pair of uplinksSDN at scale: Microsoft SDN is limited to a single cluster and does not scale beyond 16 nodes. Each cluster requires its own Network Controller, and Network Controllers do not coordinate policies, address space, or operations across clusters, which prevents centralized, multi-rack / multi-cluster SDN management.The Solution: Leaf-Spine (Clos) Topology/For disaggregated clusters, Azure Local uses a leaf-spine (Clos) topology to scale port count, increase east-west bandwidth, and reduce the blast radius of any single switch failure. Each rack has a pair of leaf switches that connect up to a shared set of spine switches. Cross-rack traffic traverses Leaf → Spine → Leaf, providing multiple equal-cost paths (ECMP) between racks. For the full explanation (including the analogy and bandwidth guidance), see the Appendix.Workload Placement ConsiderationsIn a multi-rack Clos fabric, the physical location of workloads affects traffic patterns and bandwidth requirements. While the leaf-spine topology provides equal-cost paths between any two racks, workload placement can create uneven traffic distribution that impacts performance.Key considerations:workloads with high east-west traffic between nodes — co-locating these workloads on the same rack or adjacent racks reduces spine bandwidth consumptionStorage-heavy workloads can saturate leaf-spine uplinks if spread across many racksPower-dense racks may force workload spreading across racks for thermal or power distribution reasons, increasing cross-rack trafficAzure Local does not currently provide hard workload placement controls (rack affinity). Administrators should be aware that uneven workload distribution may require adjusting the spine-to-leaf bandwidth ratio or adding spine capacity. Future guidance will be provided as workload placement tooling matures.Why Leaf’s Are the Integration Point (Not Spines)/In this architecture, the leaf layer is the smart edge (service leaf) – leaf devices handle VRF membership, VXLAN encapsulation, and anycast gateways. The service leaf pair specifically handles route leaking, DC core peering, and service integration (firewalls, load balancers).  Each cluster in the Clos architecture can be a unique VRF assignment to enable multi-tenancy for traffic segmentation. The spines are intentionally simple transit switches that only forward packets.This is a deliberate design choice:Spine Simplicity = Spine ReliabilitySpines act as the main switches for rack-to-rack communication and are designed simply—without VXLAN termination or route leaking—to reduce failures. Spine traffic operates as the underlay in a dedicated VRF, separated from cluster traffic, so misconfigurations at the spine affect all racks, while those at the leaf impact only one cluster. Leaf-to-spine connections use unnumbered interfaces for BGP peering, eliminating the need for unique /30 or /31 addresses and streamlining management.Why VRF Matters /VRF Benefits in disaggregated Azure Local ClusterVRFs (Virtual Routing and Forwarding) are essential for operational reliability and security in disaggregated clusters. They provide blast radius containment, multi-tenant safety, controlled external access, and simplified troubleshooting. VRFs ensure that routing mistakes or misconfigurations only affect isolated domains, keeping workloads secure and logically separated. Blast Radius Containment: VRF isolation prevents routing errors from impacting workload traffic, containing issues within separate routing domains.Multi-Tenant Safety: Workload types (management, compute, tenant VMs, storage) are grouped into controlled domains; new clusters get their own VRFs for isolation without extra hardware.Controlled External Access: Only the service leaf pair bridges traffic between the cluster and external networks, providing a managed boundary for north-south traffic and integrating firewalls/load balancers if needed.Simplified Troubleshooting: Issues can be quickly isolated by determining the affected VRF, reducing troubleshooting scope.Cluster-wide VRF: All racks within a cluster can use the same VRF for VM communication; underlay traffic stays separate, and internet-bound flows are managed through the service leaf, not directly from compute racks.VXLAN EVPN — The Overlay NetworkAt 64 nodes, we have multiple racks, each with its own pair of leaf switches. Because all traffic — iSCSI, CSV/Live Migration, and Cluster Heartbeat — runs over TCP, there is no requirement for lossless Ethernet. PFC is disabled on all priorities with no pause frames. Traffic is classified using 802.1p CoS tags and each class is assigned to a dedicated queue with explicit ETS bandwidth reservations enforced through Weighted Round-Robin (WRR) scheduling. iSCSI (Priority 4) receives a 50% guaranteed bandwidth reservation as the primary storage path. CSV/Live Migration (Priority 3) receives a 20% reservation to ensure adequate throughput during VM migrations. Cluster Heartbeat (Priority 7) reserves 1% — sufficient for lightweight keepalive traffic. Default traffic (Priority 0) absorbs the remaining 29% of link bandwidth. Under normal conditions all classes can burst to full line rate; the bandwidth guarantees only apply during congestion. Note that in a SAN Fibre Channel configuration, iSCSI is not present — storage traffic runs entirely on the FC fabric, separate from the Ethernet network.Note, an example of the Cisco Nexus QoS policy can be found in the Appendix QoS Configuration  Priority802.1pDescriptionPFC EnabledPauseFrameETS Bandwidth (25 GbE)ETS Bandwidth (10GbE)0Default Traffic❌ No29%28%1-2NANANANA3CSV/ Live Migration❌ No20%20%4iSCSI❌ No50%50%5-6NANANANA7Cluster Heartbeat   ❌ No1%2%Spine Leaf Switch requirementsAzure Local switch requirements for base deployments (16 nodes or fewer) are documented at Physical network requirements for Azure Local. All leaf’s connect to the shared spine switches. Traffic between racks goes: Leaf → Spine → Leaf — never more than 3 hops.Bandwidth planning: the leaf-to-spine layer is designed for a minimum 2:1 oversubscription ratio (leaf downlink bandwidth : leaf-to-spine uplink bandwidth) under normal operations.During a spine failure or maintenance window, the effective oversubscription can increase to 4:1. Congestion is expected during this period; workloads with heavy east-west traffic may see reduced throughput until full spine capacity is restored.To reduce congestion risk and improve network resiliency, the spine layer should scale horizontally as the environment grows. Adding additional spine switches increases aggregate leaf-to-spine bandwidth and adds more equal-cost paths (ECMP), improving both performance and fault tolerance.Leaf Owns the Service BoundaryBecause the service leaf pair participates in multiple VRFs (AZLOCAL as a VTEP, plus external peering), it is the natural integration point for:Route leaking — controlled import of routes between VRFs for north-south trafficFirewalls / load balancers — connected directly to service leaf ports as service appliancesVXLAN termination — wrapping and unwrapping overlay traffic for external connectivityCompute traffic leaf’s are the host edge. They terminate VXLAN for locally attached servers and provide the anycast gateways for the cluster’s workload VRF(s) (e.g., VRF AZLOCAL). They also participate in the fabric underlay to reach remote VTEPs, but they do not connect to the data-center core and they do not perform route leaking to external networks.External connectivity is centralized at the service-leaf pair, which is the dedicated boundary for DC-core peering, controlled route leaking, and service appliances (FW/LB). Keeping those functions off every compute leaf reduces the number of external touch points and limits the blast radius of a routing or policy misconfiguration to the smallest possible set of devices.The DC Core Connects Through Leaf’s, Not SpinesThe data center core network (upstream routers, firewalls, internet) does not plug directly into the spines. Instead, core connectivity is integrated at the service leaf layer — a dedicated leaf pair that handles external peering. This means:The spine layer stays pure transit — no external dependenciesCore network maintenance (firewall upgrades, router replacements) doesn’t affect the spine fabricYou can add or remove external connectivity without touching the spine configurationService appliances (firewalls, load balancers) connect at the service leaf where external peering is availableComponentRoleComplexityCompute LeafHost-facing — VRF AZLOCAL, VXLAN, anycast GWModerate (scoped to one rack)Service LeafIntegration point — route leak, DC core peering, FW/LBHigh (scoped to service pair)SpineSimple transit — forwards packets between leavesLow (fabric-wide impact if it fails)DC CoreExternal — connects via service leaf, not spineIsolated from spine fabricNoteEach cluster can be placed in its own VRF, depending on the customer's isolation requirements.This separation ensures that the spine fabric is as stable and predictable as possible. All the external integration complexity lives on the service leaf pair, and compute leaves are focused solely on serving their rack.Anycast Gateway — Why Every Switch Is the GatewayThe Old Way: HSRP/VPCIn traditional designs, two switches share a “virtual IP” (VIP) as the default gateway. One switch is “primary” and the other is “standby.” If the primary fails, the standby takes over. This works but has problems at scale:3 IPs per VLAN consumed (VIP + 2 self-IPs)Failover delay when the primary diesOnly works within a single switch pair — can’t span multiple racksThe New Way: Anycast GatewayWith VXLAN EVPN, every leaf switch is the gateway simultaneously. Every leaf switch configures the exact same IP and MAC address on each VLAN:/Source: Azure-Local-Multi-Cluster-Fabric.drawio — Tab: VRRP vs Anycast GatewayWhy this matters: A VM on any rack always has its gateway locally — on the leaf switch it’s directly connected to. No traffic needs to cross the spine just to reach the default gateway. This is critical at 64-node scale.VRF — The Concept Everyone Asks AboutWhat Is a VRF?VRF stands for Virtual Routing and Forwarding. It’s a way to create isolated routing tables inside a single physical switch — like having multiple invisible switches sharing the same hardware.The Underlay — How the Switches Talk to Each OtherWhat Is Underlay?The underlay is the physical network that connects all the switches together — the actual cables, IP addresses, and routing protocol (eBGP) that lets switches find each other. It’s the foundation that everything else (VXLAN, VRFs, EVPN) is built on top of.Think of it as the road system itself — the asphalt, lane markings, and traffic signals. The overlay (VXLAN) is the cars driving on the road. The underlay doesn’t care what’s inside the cars; it just makes sure they can get from point A to point B.How It WorksEvery switch in the fabric gets a loopback IP address — a virtual address that uniquely identifies it, independent of which physical port is used. eBGP (External Border Gateway Protocol) runs between every leaf-to-spine connection using unnumbered interfaces (no IP address on the cable itself — they use the loopback).Note: BGP unnumbered is the cleanest option, but using /31 is also fine—both are supported, so customers can choose either method.The underlay BGP sessions use IPv6 link-local addresses for peering. IPv4 routes are carried as address-family over the IPv6 BGP sessions. This design eliminates the need to assign and manage point-to-point IPv4 /31 subnets on every leaf-spine link, and it positions the fabric for future IPv6 workload support with minimal change to the underlay.The underlay runs in the default VRF (global routing table), but it is treated as infrastructure-only — it carries only loopback addresses and BGP peer routes. No default route (0.0.0.0/0) is ever injected into the underlay, and no workload prefixes are leaked into it.All workload traffic is isolated in dedicated overlay VRFs (Like Tenant, External, Storage), each mapped to its own VXLAN Network Identifier (VNI). The underlay never sees tenant or storage IP addresses — they are encapsulated inside VXLAN tunnels before entering the fabric. The underlay only forwards the outer UDP/IP headers between VTEPs.This separation is the critical security boundary. If a misconfiguration or rogue route advertisement occurs in one overlay VRF, it cannot affect the underlay or other VRFs — they are fundamentally separated by VXLAN encapsulation.Leaf-1a (Loopback: 10.255.1.1, ASN 65001)   │  eBGP unnumbered (100G) — Default VRF   ▼Spine-1 (Loopback: 10.255.0.1, ASN 65000)   │  eBGP unnumbered (100G) — Default VRF   ▼Leaf-3a (Loopback: 10.255.3.1, ASN 65003)Each rack gets its own unique ASN (Autonomous System Number) — Rack 1 is 65001, Rack 2 is 65002, etc. The spines share ASN 65000. This means every leaf-to-spine connection is an eBGP session (external BGP, between different ASNs), which gives each rack routing independence.Traffic Flow: East-West vs North-SouthThe two most common questions: “How does Rack 1 talk to Rack 3?” and “How does a VM reach the internet?”/Source: Azure-Local-Multi-Cluster-Fabric.drawio — Tab: East-West and North-South TrafficFlowPathVRF BoundaryEast-West (Rack 1 → Rack 3)VM → Leaf-1 (VXLAN wrap) → Spine → Leaf-3 (VXLAN unwrap) → VMStays inside VRF AZLOCAL the entire pathNorth-South (VM → Internet)VM → Compute Leaf (VXLAN wrap) → Spine (transit) → Service Leaf (VXLAN unwrap, route leak) → DC Core → InternetCrosses VRF boundary at the service leafInter-VLAN Routing — Can VLAN 7 Talk to VLAN 6?Yes. It never inspects the inner payload.Egress leaf decapsulates the VXLAN header in VRF AZLOCAL and delivers to the destination VM.The VM payload travels inside a VXLAN tunnel. The spine only sees the outer UDP header and routes it based on underlay VRF entries. The two VRFs — AZLOCAL (workload) and underlay (infrastructure) — are completely isolated from each other.Why This MattersPropertyEffectVRF AZLOCAL has no default routeVMs cannot accidentally route to the internet — north-south requires an explicit route leak at the service leafUnderlay VRF has no default routeSwitch-to-switch communication cannot be redirected externally — prevents route poisoningSpine sees only underlay VRFA compromised spine cannot inspect or inject workload trafficNo route leaking on compute leavesCompute leaves only participate in VRF AZLOCAL — no external routes are presentHow VRF Looks on the Switch/Where Does 0.0.0.0/0 Live?The default route (0.0.0.0/0) the path to the internet — does not exist in the underlay VRF. The underlay is intentionally isolated from external routing to protect it. No default route is leaked into the underlay, and no traffic from the underlay can reach the internet. This prevents route poisoning — if an attacker or misconfiguration injected a default route into the underlay, it could hijack all switch-to-switch communication.Underlay VRF:  Loopback routes + BGP peer routes ONLY               ❌ No 0.0.0.0/0 — no path to the internet               ❌ No leaked routes from any other VRFVRF AZLOCAL:   Workload routes only               ❌ No 0.0.0.0/0 by default — VMs can't reach the internetNeither VRF has a default route by default. This is the most secure starting position — nothing can reach the internet unless you explicitly configure it.To enable north-south (internet) access, there are two options — both happen at the service leaf, never the spine or compute leaf:Route leak on the service leaf — The service leaf pair participates in VRF AZLOCAL (as a VTEP) and has external peering to the DC core. It performs a route leak to bridge AZLOCAL traffic to the DC core. VM traffic travels from the compute leaf through the spine to the service leaf via VXLAN, then exits through the DC core peering.Integrated firewall / load balancer — A firewall or load balancer is connected to the service leaf fabric as a service appliance. Traffic is steered to the appliance for inspection before exiting to the internet.In both cases, the spines remain transit-only (underlay VRF, no default route). Compute leaves only run VRF AZLOCAL — they have no external routes or DC core peering. External connectivity is introduced at the service leaf — the integration point.Quick Reference: VRF FAQQuestionAnswerCan Azure Local traffic accidentally leak to the internet?No — it’s locked inside VRF AZLOCAL. VRF separates traffic at Layer 3 (IP routing). VLANs are members of a VRF.Why not just use firewall rules instead of VRFs?Firewall rules can be misconfigured or removed.
